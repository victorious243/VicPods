# VicPods DSAR Workflow

Internal workflow for handling privacy and data-rights requests. Practical guidance only.

## Request Intake

Accepted channels:

- `PRIVACY_CONTACT_EMAIL`
- Help Center / support workflow when routed to privacy handling

## Minimum Information To Request

- account email
- the right being exercised
- enough detail to identify the request scope

Do not ask for unnecessary data.

## Identity Verification

Before acting on sensitive requests, confirm identity proportionately. Examples:

- signed-in request from the same account session
- reply from the same account email
- password re-entry or similar account-level confirmation where appropriate

## Request Types

### Access

- confirm identity
- identify systems holding data
- gather user-facing account and product data
- exclude data that must not be disclosed to another person

### Rectification

- check whether the issue can be solved through Settings
- if not, correct data in the relevant system and log the change

### Erasure

- check whether self-serve deletion already covers the request
- review whether any finance, fraud-prevention, or legal-retention exception applies
- confirm what was deleted and what was retained, with reason

### Restriction / Objection

- identify the processing activity
- assess lawful basis and whether the objection can be honoured
- document outcome clearly

### Portability

- export applicable user-provided data in a reasonable structured format where the right applies

## Response Timing

- target: within 1 month
- if extended, explain why and record the new deadline

## Logging

For each request, log:

- request date
- requester identity
- request type
- systems checked
- decision
- response date
- retention exception, if any

## Escalation

Escalate immediately if:

- identity is uncertain
- the request involves security incidents
- the request may affect another person’s data
- a supervisory authority complaint has been mentioned
